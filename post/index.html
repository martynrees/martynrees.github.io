    Meraki PCAP Analyser

    Aside from the Proactive PCAP feature, this would have to be the best addition to the Meraki dashboard in recent times. Not only can we view a PCAP thats been taken, but we can also get some auto analysis (Meraki Assurance) on what the potential problem may be. This is excellent for the seasoned wifi professional, but even more valuable for a inexperienced engineer as not only can it guide them to the issue, they can also learn from the problems at the same time. Landing on the analysis screen we are given an overview of the problem with the option to expand out the relevant sections for more detail. We can also view the PCAP in a wireshark like view straight from the dashboard as well, so this saves the hassle of downloading and opening the file for just a quick look. Given that the anlysis has already been done, we are given a pretty good guide that in this instance we had a problem with DHCP offers not being received by the client. This gives us a quick answer, but sometimes we might still want to dig in to what went on and actually investigate the flow of frames. For this we can expand out the packet capture to view the frames just as we would in Wireshark. I’ve selected a bit more interesting capture for this exercise. From here we are presented with a default view or profile, which is ok, but not great. So down the bottom right hand corner of the panel we can adjust the profile depending on the scenario of the capture. Selecting the OTAWireless profile we are shown the more relevant columns, and like most wireless engineers prefer, we get colour coded Frame Types. Just like in Wireshark, we can then create filters. For a proactive PCAP this might not be as useful, but for a manual capture that you’ve then selected to store in the cloud this is of more use. If we are investigating issues with retries then we can filter with wlan.fc.retry == True to filter all of the frames that are being retransmitted. At this stage if we want to get the statistics view then we would still need to download and open the file in Wireshark, but hopefully this feature will be added down the track. We can also inspect the frames from within the dashboard as well to see things like the Radiotap header, 802.11 radio information, Wireless Management Information Elements, etc, so for a lot of cases this speeds up the analsysi process, and it also doesn’t matter if you are using a machine that doesn’t have Wireshark installed, such as a jump host or similar. Whilst there are more features coming, and it’s not GA yet, the PCAP analsysis and Wireless Assurance is an excellent addition to the dashboard that really enahnced Meraki’s capabilities to reduce the complexities of wireless. ...

    Meraki Proactive Packet Capture

    The Proactive Packet Capture is a tool that I’ve used for a long time now in Catalyst Center (formally DNA-C) so it’s exciting that it’s now been introduced to the Meraki side of the world as well. With over 100 event triggers that can occur during the client onboarding process or normal operations, there is a fair chance that whatever issue a client is experiencing in the network that it will be captured automatically. The Proactive PCAP does require MR31 or higher, and the APs need to be Wifi 6/6E or 7 to support the feature. The retention time for proactive captures is 7 days, so when enabling the feature you do need to decide if you want to enable it for some or all devices. You can select which APs you enable, which if you have a large/busy network would probably be the better way to go. Since I’m running this at home with only a couple of APs I’m happy to turn it on for all devices. The other thing to note is that once this feature is GA you will need a A MR-ADV (Advanced) license to use the Proactive PCAP feature. This will be enforced in the future. As this is for my home, I had to leave it to run for a few days and I came back to find plenty of captures, most weren’t ones to worry about as they were ones like EAPoL timeout’s that happened as we were leaving the house, but there are certainly real world examples of where this would be important. Looking down the right hand side of the page we can see which step of the onbaording process the trigger occured at (Authentication, Assosciation, DHCP, etc) and we have the option to analyse the capture directly on the dashboard, or download the pcap to review in Wireshark on our local machine. These options are available regardless of if the capture was proactive or manually run. In the next post we’ll have a look at the analysis view that we get on the dashboard. Again though, this is another instance where the smarts of the product can help save time and effort by capturing events before you even know that they’ve happened, and can save many hours trying to recreate problems or disrupting users even further to try and reproduce. ...
